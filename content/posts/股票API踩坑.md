## 起因

博客搭起来之后总觉得少了点什么。首页像个名片，随笔列表空荡荡的。我就想着加一个「股票观察」的页面，放一些指数和自选股数据，打开博客就能看到行情，不管实用不实用，权当装饰。

数据源选了新浪财经的免费 API。不用申请 key，直接 HTTP 请求就行，对个人项目来说再合适不过了。

## 第一个问题：数据源选哪个

一开始搜「免费股票 API」，出来的结果还挺多的。新浪、腾讯、东方财富，都有接口。

最后选了新浪，原因是：
- 不用注册、不用 key
- 接口简单，直接 GET 就完事了
- 十几年的老牌来源，数据还算靠谱

腾讯的 API 试了一下，返回的数据格式比新浪的复杂，而且有些字段名很奇怪，就放弃了。

## 第二个问题：有些指数数据是错的

代码写好了，跑起来一看——上证指数和创业板是对的，但标普 500 显示 **6643 点**，纳斯达克显示 **22484 点**。

我：？这两个数字看着不太对啊，标普 500 不是应该在 7000 多吗？

查了半天，发现是用错了新浪 API 代码。新浪里 `int_sp500` 这个代码返回的**不是标普 500**，是某个 S&P 的变种指数。纳指也是类似的问题——`int_nasdaq` 返回的是纳斯达克综合指数，而不是纳斯达克 100。

新浪里真正的美股指数要用 `gb_` 前缀：

```
以前:
  int_sp500   → 标普指数 6643  ← 不是标普500
  int_nasdaq  → 纳斯达克 22484 ← 这是综合指数

改后:
  gb_inx      → 标普500指数 7383  ✅
  gb_ndx      → 纳斯达克100 28957 ✅
```

而且这两种代码返回的数据格式还不一样，字段位置不同。`int_` 的涨跌幅在 fields[3]，`gb_` 的在 fields[2]。写解析代码的时候差点给我绕进去。

```java
// 不同前缀的字段位置完全不一样，得分开处理
if (isUS) {
    changePct = parseDouble(fields[2]);  // gb_ 前缀
} else {
    changePct = parseDouble(fields[3]);  // int_ / s_ 前缀
}
```

## 第三个问题：美股没开盘就显示 0%

数据调对了之后上线跑了两天，有一天突然发现一个问题——**一到周末或者晚上，美股自选股的涨跌幅全变成 0%**。

一开始以为代码又写错了，查了了半天反应过来：美股还没开盘，新浪在休市的时候会把最新价和昨收价都设成同一个值，算出来的涨跌自然就是 0%。

但这个体验确实不好。用户（虽然也没别人）打开页面想看「昨天微软跌了多少」，结果显示 0%，什么信息都没有。

最后的解决方案是**加了个内存缓存**：每次涨跌幅不为 0 的时候就存下来，下次如果发现涨跌幅是 0（可能是休市了），就返回缓存里的值。

```java
// 休市时兜底：用缓存的上次非零值
if (changePct == 0.0 && lastChangeCache.containsKey(code)) {
    changePct = lastChangeCache.get(code);
} else if (changePct != 0.0) {
    lastChangeCache.put(code, changePct);
}
```

当然这个方案有个小问题：如果后端重启了，缓存就没了，又显示 0%。不过等美股开盘之后数据就回来了，缓存也会重新填充。不是什么大事。

## 最后一个坑：硬编码的降级数据

最早版本的代码里，如果新浪 API 挂了，会返回一组**手写死的假数据**。

当时觉得这是一个挺好的「兜底方案」——万一 API 挂了，至少页面不报错。但某天打开页面查看的时候，意识到页面在展示错误数据。标普 500 实际已经涨到 7000 多，降级数据还显示着 5302。

改成了**直接报错**：API 挂了就让前端显示错误信息。比「看起来正常工作但其实数据是错的」好得多。

## 总结

接入外部数据源真的是各种小坑。现在看着大盘数据正常刷新，还是挺有成就感的。

接下来想把日经 225 也加上（新浪好像不支持，得换个数据源），自选股的卡片样式也想重新设计一下，后面打算直接接TradingView的。

*如果有更好的免费股票数据源，欢迎推荐。新浪这个虽然能用，但确实有些局限性。*
